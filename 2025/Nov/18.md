# Learning Log: Importing a PostgreSQL Database into GCP CloudSQL

## Summary

Today was about learning to set up a PostgreSQL instance in Google Cloud Platform (GCP) and importing a database dump file containing tables and data.

---

## Steps and Experiments

### 1. Starting Fresh  
- Logged into the GCP console after a long time.  
- Deleted old projects from the IAM page.  
- Created a new project from scratch.  

### 2. Creating the CloudSQL Instance  
- Navigated to CloudSQL.  
- Created a PostgreSQL 18 instance (`jaydb18`) - kept it sandbox/basic.  
- Created a new database called `hospital` (used for the IDC 21 day SQL challenge).

### 3. Dump File Import Issues  
- Noticed GCP doesn’t allow direct import of `.dump` files.  
- Tried renaming `.dump` to `.sql` – import failed.  
- Googled and found the following command to convert dump to SQL:
pg_restore -f "C:\Users\Jay\OneDrive\Jay Study\SQL\IDC 21 day SQL challenge\hospital_beds_server.sql" "C:\Users\Jay\OneDrive\Jay Study\SQL\IDC 21 day SQL challenge\hospital_beds_server.dump"
- Still no success importing the file.

### 4. Manual Table Creation and Data Import  
- Manually created tables using SQL scripts from the `.sql` file.
- The insert scripts were poorly formatted – switched to using CSV files for the data.
- Initial CSV import failed due to header row; removed header manually and import succeeded.

### 5. Storage Bucket Setup and Permissions  
- Created a Google Cloud Storage bucket (since imports must use bucket files, not direct local uploads).

### 6. Restart and Improved Approach  
- Wasn’t satisfied with the step-by-step fixes, so restarted the whole process.
- Found this guide helpful:  
[CloudSQL Import/Export Docs](https://docs.cloud.google.com/sql/docs/postgres/import-export/import-export-sql)
- Added and edited custom IAM roles with all necessary permissions.

### 7. Troubleshooting  
- Deleted unnecessary users after encountering errors.
- Restarted the DB instance.
- Found the option to select the user for import on the import screen.
- Selected `postgres` user, retried the import, and it worked!

---

## Success

- Successfully imported the database and could query the data in CloudSQL.

---

## Lessons Learned

- CloudSQL import requires `.sql` (plain SQL text), not `.dump` (binary).
- Use `pg_restore` or other conversion methods for compatible format conversion.
- Manual intervention (table creation, header removal) might be needed with imperfect files.
- Storage buckets are mandatory for imports.
- IAM permissions are crucial—double-check roles!
- The user selection in import UI is essential for permissions.
